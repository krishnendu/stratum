# Phase 8 — Android build (`stratum-mobile-core`)

This document describes the local build pipeline that produces the
`libstratum_mobile_core.so` artifact bundled inside the Android app, plus
the Kotlin bindings the app uses to call into it.

The Rust side lives in [`crates/stratum-mobile-core`](../crates/stratum-mobile-core/).
The crate is a `cdylib` (plus `rlib` for in-workspace tests) that exposes a
deliberately tiny C-ABI surface generated by [`uniffi`](https://mozilla.github.io/uniffi-rs/).

## Status

Scaffold only. The current surface is two stub functions:

| Symbol            | Returns | Real behaviour lands in           |
|-------------------|---------|------------------------------------|
| `mobile_init()`   | `u32`   | `feat/phase-8-mobile-core`         |
| `mobile_version()`| `String`| now (returns workspace version)    |

The UDL contract lives in [`crates/stratum-mobile-core/src/stratum.udl`](../crates/stratum-mobile-core/src/stratum.udl)
and is the source of truth for what the Kotlin / Swift sides see.

## Prerequisites

- Rust toolchain pinned by `rust-toolchain.toml` (currently `1.90`).
- Android NDK r26+ on `PATH` (provides `aarch64-linux-android` linker).
- The `aarch64-linux-android` Rust target installed:
  ```
  rustup target add aarch64-linux-android
  ```
- `uniffi-bindgen` matching the `uniffi` version pinned in the workspace
  (`0.28.x`). Install once with:
  ```
  cargo install --version 0.28.3 uniffi_bindgen
  ```
- A `~/.cargo/config.toml` block that points cargo at the NDK linker, e.g.:
  ```toml
  [target.aarch64-linux-android]
  linker = "/path/to/ndk/toolchains/llvm/prebuilt/<host>/bin/aarch64-linux-android26-clang"
  ar     = "/path/to/ndk/toolchains/llvm/prebuilt/<host>/bin/llvm-ar"
  ```

## Build steps

1. Cross-compile the `cdylib` for arm64 Android:

   ```
   cargo build --release --target aarch64-linux-android --lib -p stratum-mobile-core
   ```

   Output: `target/aarch64-linux-android/release/libstratum_mobile_core.so`.

2. Generate the Kotlin bindings from the UDL:

   ```
   uniffi-bindgen generate crates/stratum-mobile-core/src/stratum.udl \
       --language kotlin \
       --out-dir generated/kotlin
   ```

   Output: `generated/kotlin/uniffi/stratum_mobile/stratum_mobile.kt`.

3. Drop both artifacts into the Android module:
   - `libstratum_mobile_core.so` → `app/src/main/jniLibs/arm64-v8a/`
   - `stratum_mobile.kt`        → `app/src/main/java/uniffi/stratum_mobile/`

## Future targets

The Android app is the Phase-8 driver, but the same `cdylib` is the iOS
boundary too. The matrix we plan to ship:

| Target                       | Use            |
|------------------------------|----------------|
| `aarch64-linux-android`      | Android arm64  |
| `armv7-linux-androideabi`    | Android armv7  |
| `x86_64-linux-android`       | Android x86\_64 emulator |
| `aarch64-apple-ios`          | iOS device     |
| `aarch64-apple-ios-sim`      | iOS simulator (Apple Silicon) |
| `x86_64-apple-ios`           | iOS simulator (Intel)   |

The matrix lives here so the eventual `xtask mobile` driver has one place
to enumerate against.

## Why uniffi (and not hand-written JNI)

- One UDL definition produces both Kotlin and Swift bindings.
- The generator emits the `JNI_OnLoad`-style boilerplate, error mapping,
  and `Pointer` lifetime management we'd otherwise hand-write per call.
- It already handles the cross-language `Result<T, E>` mapping that we'd
  need for `StratumError` once the full runtime is wired in
  (`feat/phase-8-mobile-core`).

## Workspace-lint exceptions

`crates/stratum-mobile-core/src/lib.rs` carries two crate-level `#[allow]`
attributes with `reason = "..."`. They exist because the
`uniffi::include_scaffolding!` macro expands into generated code we don't
author, which trips:

- `unsafe_code` — the scaffolding emits `#[no_mangle] extern "C"` shims.
- `clippy::missing_const_for_fn` / `clippy::empty_line_after_doc_comments`
  — fired against the generated file in `target/`, which we cannot annotate
  locally.

No hand-written `unsafe` blocks live in this crate.
